# IEC 61131-3 Sequential Function Chart (SFC) Reference

This document explains the key concepts of IEC 61131-3 Sequential Function Charts and how they are represented in this editor.

## 1. What is an SFC?

A Sequential Function Chart (SFC) is a graphical language for modeling sequential control systems. An SFC consists of:

- **Steps**: Basic states or control points.
- **Transitions**: Logical conditions that determine movement between steps.
- **Actions**: Work executed while a step is active.
- **Branches**: Alternative or simultaneous execution paths.
- **Initial step**: The starting active step when the chart begins.

The SFC is executed in scan cycles: evaluate transition conditions, deactivate/activate steps, and execute actions for active steps.

## 2. Core IEC SFC concepts

### Steps

- Represent states or phases of execution.
- A step can contain one or more actions.
- One step is marked as the **initial step**.
- In this editor, the initial step is drawn with a double border.

### Transitions

- Represent conditions that must be true to move from one step to the next.
- In an SFC, transitions typically connect steps in a vertical sequence.
- A transition can have a condition expression such as `TRUE`, `X1 AND NOT X2`, or any valid PLC expression.

### Actions and qualifiers

Actions are associated with steps, and qualifiers define when the action is executed.

Supported qualifiers in this editor:

- `N` — Non-stored: execute while step is active
- `S` — Set: set an output when the step becomes active
- `R` — Reset: clear an output when the step becomes active
- `P` — Pulse: generate a pulse once when the step becomes active
- `D` — Delay: execute after a configured time delay
- `L` — Timed: execute for a configured duration
- `SD` — Set then delay (vendor-specific)
- `DS` — Delay then set (vendor-specific)
- `SL` — Set and hold for time (vendor-specific)
- `P1` — Pulse on activation (vendor-specific)
- `P0` — Pulse on deactivation (vendor-specific)

Qualifiers `L`, `D`, `SD`, `DS`, and `SL` require a time interval value such as `T#10s`.

### Branching

Two branch styles are supported:

- **Alternative branches**: Only one path is taken. Represented by a single horizontal synchronization bar.
- **Simultaneous branches**: Multiple paths execute in parallel. Represented by a double horizontal synchronization bar.

Branch nodes have a role:

- `divergence`: branch start
- `convergence`: branch end

Branch lanes are ordered arrays of SFC nodes. The chart structure is nested through lanes.

## 3. App-specific model mapping

This editor uses a JSON-based data model in `src/model/types.ts`.

### SFC diagram root

```ts
export interface SfcDiagram {
  version: 1;
  title: string;
  root: SfcNode[];
}
```

### Supported nodes

```ts
export type SfcNode = SfcStep | SfcTransition | SfcBranch;
```

#### SfcStep

```ts
export interface SfcStep {
  id: string;
  kind: 'step';
  name: string;
  isInitial?: boolean;
  actions: SfcAction[];
}
```

- `id`: unique node identifier
- `name`: step label
- `isInitial`: true for the initial step
- `actions`: attached action list

#### SfcTransition

```ts
export interface SfcTransition {
  id: string;
  kind: 'transition';
  condition: string;
}
```

- `condition`: transition expression

#### SfcBranch

```ts
export interface SfcBranch {
  id: string;
  kind: 'branch';
  branchType: BranchType;
  role: BranchRole;
  label?: string;
  lanes: SfcNode[][];
}
```

- `branchType`: `alternative` or `simultaneous`
- `role`: `divergence` or `convergence`
- `lanes`: arrays of `SfcNode` sequences, one per branch path

### Actions

```ts
export interface SfcAction {
  id: string;
  name: string;
  qualifier: ActionQualifier;
  time?: string;
}
```

- `name`: action identifier or output name
- `qualifier`: execution mode
- `time`: optional time interval when required

### Helper types

```ts
export type BranchType = 'alternative' | 'simultaneous';
export type BranchRole = 'divergence' | 'convergence';
export type ActionQualifier = 'N' | 'S' | 'R' | 'P' | 'D' | 'L' | 'SD' | 'DS' | 'SL' | 'P1' | 'P0';
```

Utility guards are available in `src/model/types.ts`:

- `isStep(node)`
- `isTransition(node)`
- `isBranch(node)`
- `formatAction(action)`

## 4. JSON file format used by the editor

Diagrams are saved as `.sfc.json` files. The minimum structure is:

```json
{
  "version": 1,
  "title": "MySequence",
  "root": [
    {
      "id": "step-...",
      "kind": "step",
      "name": "Init",
      "isInitial": true,
      "actions": []
    }
  ]
}
```

### Branch example

```json
{
  "id": "branch-...",
  "kind": "branch",
  "branchType": "alternative",
  "role": "divergence",
  "label": "JumpLabel",
  "lanes": [
    [ { "id": "t1", "kind": "transition", "condition": "X" }, { "id": "s1", "kind": "step", "name": "A", "actions": [] } ],
    [ { "id": "t2", "kind": "transition", "condition": "Y" }, { "id": "s2", "kind": "step", "name": "B", "actions": [] } ]
  ]
}
```

Each branch lane is a sequence of steps, transitions, or nested branches.

## 5. Validation and editor behavior

The editor validates diagrams using `src/validation/iecValidator.ts`.

### Required structure rules

- `root` must contain at least one element.
- Diagram must start with a `step` node.
- Steps and transitions should alternate in a linear sequence.
- Branch nodes may appear instead of a plain step/transition sequence.
- Exactly one step must have `isInitial: true`.
- Step names must be non-empty and unique.

### Actions

- Action names must be non-empty.
- Qualifiers `L`, `D`, `SD`, `DS`, and `SL` require a `time` value.
- Vendor-specific qualifiers (`SD`, `DS`, `SL`, `P1`, `P0`) are accepted, but their semantics may vary across PLC implementations.

### Transition conditions

- Empty transition conditions raise a warning.
- Use meaningful logical expressions for branch decisions.

### Branch validation rules

Alternative branch lanes:

- Divergence lanes must start with a `transition`.
- Convergence lanes must end with a `transition`.

Simultaneous branch lanes:

- Divergence lanes must start with a `step`.
- Convergence lanes must end with a `step`.

All branch lanes must have at least two lanes.

### Reachability and diagnostics

- The editor reports unreachable elements as warnings.
- It also reports missing or duplicate initial steps.

## 6. How this editor renders SFC elements

- **Initial step**: double-bordered rectangle.
- **Step**: standard rectangle with step name.
- **Transition**: horizontal bar with condition text.
- **Action**: attached text block formatted as `QUALIFIER Name(time)`.
- **Alternative branch**: single synchronization bar.
- **Simultaneous branch**: double synchronization bar.

Nested branches can be represented inside branch lanes.

## 7. Usage notes for developers and AI agents

- Prefer the `src/model/types.ts` data model as the canonical source of truth.
- Use `src/validation/iecValidator.ts` for rule enforcement and expected errors.
- The `examples/` directory contains valid `.sfc.json` diagrams demonstrating linear, alternative, simultaneous, and nested patterns.
- When modifying the editor, update both the renderer and validator to keep visualization and semantics aligned.

## 8. Example files

The repository includes example diagrams in `examples/`:

- `linear.sfc.json`
- `alternative.sfc.json`
- `simultaneous.sfc.json`
- `nested.sfc.json`

These are useful references for how branch lanes and node ordering should be structured.

## 9. Important constraints for this project

- The editor does not currently support arbitrary graph cycles outside the structured SFC sequence.
- Branch lanes must follow the type-specific lane start/end requirements.
- Actions are attached only to steps, not transitions or branch nodes.
- The JSON model is strongly typed around `version: 1` and a root node array.
